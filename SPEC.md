# SPEC.md — 亲子任务打卡 & 奖励系统

> **项目名称**：小小动物园 · 亲子任务打卡系统  
> **版本**：v1.0  
> **日期**：2026-07-07

---

## 1. 问题陈述

### 1.1 要解决什么问题

家中有小学/幼儿园阶段的孩子，家长希望：
- 培养孩子良好的日常习惯（读书、写字、练琴等）
- 让孩子理解"努力就有回报"的概念
- 减少家长与孩子之间关于"做没做"、"能不能玩"的拉扯
- 让亲子互动更有趣、更有仪式感

### 1.2 目标用户

- **主要用户**：6-10 岁儿童（幼儿园中大班 ~ 三年级）
- **管理用户**：孩子的父母/监护人
- **每个家庭独立使用**，孩子间数据不共享

### 1.3 为什么值得做

将"习惯养成"与"游戏化"结合，以"动物园员工"的主题包装积分系统，配合盲盒机制、宠物养成、数学游戏等元素，让孩子在趣味中培养自律。家长端提供完整的配置与管理能力，同时"美好时刻"功能为家庭留下温暖的记忆记录。

## 2. 用户故事

> 遵循 INVEST 原则（Independent, Negotiable, Valuable, Estimable, Small, Testable）

1. **作为孩子**，我希望每天看到今天有哪些任务可以做，打卡后能立即看到我获得了多少积分和金币，这样我有动力去完成任务。

2. **作为孩子**，我希望去积分商城浏览可以用积分兑换的活动，兑换后能看到家长上传的照片证明真的带我去了，这样我觉得公平透明。

3. **作为孩子**，我希望养一只属于自己的宠物，给它喂食、玩耍、摸它，每次互动都有不同的反应，这样即使一个人也能获得陪伴感。

4. **作为孩子**，我希望和宠物玩数学游戏，答对题目宠物会开心，答错也能学到正确答案，这样学习变得有趣。

5. **作为孩子**，我希望看到我获得的勋章墙和心愿单进度，这样我能感受到自己在不断进步。

6. **作为家长**，我希望可以管理任务列表（增删改），设置每个任务的基准积分、波动范围，以及是否锁定积分。

7. **作为家长**，我希望可以管理兑换活动，设置活动名称、积分价格、是否需要上传照片，并按需启用或停用。

8. **作为家长**，我希望每次带孩子完成兑换的活动后上传照片，记录"美好时刻"，让全家人看到。

9. **作为家长**，我希望每周看到孩子的成长报告（任务完成率、积分变化、勋章获得等），了解孩子的近况。

10. **作为家长**，我希望可以修改家长自己和孩子的账号密码，设置孩子所在年级（影响数学题难度）。

## 3. 功能规约

### 模块 A：任务打卡系统（核心模块）

| 项目 | 内容 |
|------|------|
| **输入** | 孩子选择当天的任务，点击"打卡" |
| **行为** | 系统按盲盒机制计算实际积分：`实际分 = 基准分 + 随机波动(下限~上限)`；同时产生金币 = `实际分 × 10` |
| **输出** | 打卡后揭晓积分和金币数，显示庆祝动画文案 |
| **边界条件** | 同一天同一任务不可重复打卡；可补打前一天的任务 |
| **错误处理** | 如果任务已被家长停用，前端隐藏不显示 |

**盲盒机制细节：**
- 孩子在打卡前只看到任务名和"？"符号
- 打卡后揭晓实际分数和金币
- 基准分由家长设置，波动范围由家长设定上下限
- 锁定状态的任务不波动，固定给基准分

**园长每周总结（基准分调整）：**
- 每 5-7 天自动触发一次
- 完成率 ≥ 80%：基准分 +1（"园长很高兴，给你涨工资！🎉"）
- 完成率 ≤ 40%：基准分 -1（"园长有点难过，下周加油赚回来！💪"）
- 区间内：基准分不变（"园长说你表现还不错，继续保持！"）
- 调整后的基准分不可超出家长设定的全局上下限

**预设任务：** 读书、写字、画画、弹钢琴、跳舞、做数学题。家长可新增、删除、修改任意任务。

### 模块 B：积分商城 & 兑换系统

| 项目 | 内容 |
|------|------|
| **输入** | 孩子从商城列表中选择活动，确认兑换 |
| **行为** | 扣除相应积分，记录兑换，生成待完成的兑换单 |
| **输出** | 显示兑换成功，扣除积分，金币不变 |
| **边界条件** | 积分不足时提示"积分还不够哦，继续加油！" |
| **错误处理** | 活动被家长停用则不再显示 |

**活动类型：**
- 时间类（看手机 X 分钟）：积分消耗 = 单价 × 分钟数
- 单次类（吃好吃的 / 买小玩具 / 出门游玩）：固定积分消耗

**美好时刻（照片上传）：**
- 高积分活动需要家长上传照片作为凭证
- 家长上传照片后，孩子端能看到"爸爸妈妈带我去了……"的记录
- 不上传不影响积分扣除，但会标记"待上传"
- 不上传 OCR 识别，仅作存储展示

### 模块 C：家长管理端

| 项目 | 内容 |
|------|------|
| **输入** | 家长登录后进入管理界面 |
| **行为** | 管理任务、活动、查看报告、管理账号等 |
| **输出** | 配置保存后实时生效 |

**功能列表：**
1. 首次运行引导创建管理员账号
2. 任务管理：增删改任务、基准分、波动范围、锁定
3. 活动管理：增删改兑换活动、积分价格、是否需要照片
4. 孩子管理：创建孩子账号、修改密码、设置年级
5. 账号管理：修改自己的密码
6. 成长报告：查看孩子的周统计数据
7. 美好时刻墙：查看所有上传的照片
8. 勋章管理：查看孩子获得的勋章

### 模块 D：宠物养成系统

| 项目 | 内容 |
|------|------|
| **输入** | 孩子选择互动类型（喂食/玩耍/摸一摸/数学游戏） |
| **行为** | 扣除金币，增加成长值，显示对应的互动文案 |
| **输出** | 宠物反应文案 + 成长进度更新 |

**宠物机制：**
- 同时只养 1 只，养大后才能领第 2 只，最多累计 5 只
- 蛋阶段：仅需喂食（8 次）→ 孵化
- 幼年阶段：喂食（15 次）+ 玩耍（10 次含数学游戏）→ 成年
- 成年后可放生，放生的宠物偶尔会回来"看望主人"
- 成长完全按操作次数，不受真实时间影响

**5 种动物预设：** 小猫、小狗、小兔子、小恐龙、小企鹅（每种有 15 条不同的互动文案）

**日常消耗：**
- 日常喂食 20 金币/次
- 清洁洗澡 30 金币/次
- 偶尔生病 100 金币/次

| 物品 | 价格 |
|------|:----:|
| 宠物粮（蛋期+幼年期） | 50 金币/次 |
| 小玩具（幼年期玩耍） | 80 金币/次 |

### 模块 E：数学游戏

| 项目 | 内容 |
|------|------|
| **输入** | 孩子选择"和宠物玩数学游戏"，系统出题 |
| **行为** | 显示题目 → 孩子输入答案 → 判断对错 |
| **输出** | 答对：宠物开心动画 + 额外成长值；答错：显示正确答案 |

**难度分级（家长设置）：**

| 年级 | 题目范围 |
|------|---------|
| 幼儿园 | 10 以内加减法 |
| 一年级 | 100 以内加减法，无进位退位 |
| 二年级 | 九九乘除法 + 100 以内加减法（有进位退位） |
| 三年级 | 1000 以内加减法 + 有余数除法 + 一位/两位数 × 整十整百数 |

### 模块 F：勋章墙

| 勋章 | 获得条件 |
|------|---------|
| 🌟 初来乍到 | 完成第 1 个任务 |
| 🌟 坚持不懈 | 连续 7 天完成任务 |
| 🌟 数学小天才 | 数学题累计答对 20 题 |
| 🌟 攒钱小能手 | 累计获得 500 积分 |
| 🌟 宠物达人 | 成功养大 3 只宠物 |
| 🌟 心想事成 | 心愿单完成 1 个目标 |
| 🌟 满分一周 | 本周任务完成率 100% |

### 模块 G：成长报告

- 每周自动生成（也可手动查看）
- 展示内容：任务完成率、积分/金币收支、宠物状态、勋章获得
- 以简单图表形式呈现（HTML 页面）

### 模块 H：心愿单

- 孩子可以添加心愿（想要的东西/想做的事）
- 设置目标积分，显示进度条
- 达到目标后标记为"已完成"
- 家长可以看到孩子的心愿

## 4. 非功能性需求

### 4.1 性能
- 页面加载时间不超过 2 秒
- 同时支持 1-3 个家庭成员使用

### 4.2 安全
- 所有密码使用哈希加密存储
- 敏感配置通过环境变量加载，绝不硬编码进源码
- 密钥/凭据绝不提交 Git 仓库

### 4.3 可用性
- 界面简洁，孩子端以大按钮和图标为主
- 家长端功能分区清晰
- 首次运行有初始化引导

### 4.4 可观测性
- 所有关键操作记录日志
- AGENT_LOG.md 记录开发过程中的关键决策

                      │

### 数据流

```
家长创建任务 → 孩子看到任务列表 → 孩子打卡 
→ 积分引擎计算盲盒分数 → 记录积分+金币 
→ 孩子用积分兑换活动 → 家长上传照片 
→ 积分扣除，金币用于养宠物
```

## 7. 凭据与分发设计

### 7.1 凭据安全

- **密码存储**：使用 Werkzeug 的 `generate_password_hash` 进行哈希加密，不存明文
- **API Key / 敏感配置**：通过环境变量加载，或在首次运行时引导用户录入
- **.env 文件**：用于本地开发环境，添加至 `.gitignore` 确保不提交
- **Git 安全**：提交前自查，确保仓库中无任何明文凭据

### 7.2 分发方案

**方案一：Docker 容器（满足作业要求）**
- 提供 `Dockerfile`，一条命令构建运行
- CI（GitHub Actions）自动构建 Docker 镜像

**方案二：云端部署（满足作业要求）**
- 部署到 Render.com（免费，无需信用卡）
- 提供公网访问 URL

**用户使用流程：**
1. 打开浏览器 → 访问链接
2. 首次运行 → 自动跳转到"创建管理员账号"
3. 创建账号后进入家长端 → 创建孩子账号 → 配置任务
4. 孩子登录 → 开始每日打卡

## 8. 技术选型与理由

| 技术 | 选型 | 理由 |
|------|------|------|
| **后端框架** | Python Flask | 轻量、上手快，适合小型 Web 应用 |
| **数据库** | SQLite | 零配置，单文件，Python 内置支持 |
| **前端** | HTML + CSS + JavaScript | 无需学习前端框架，纯静态页面 |
| **模板引擎** | Jinja2（Flask 内置） | 服务端渲染，简化前后端交互 |
| **密码加密** | Werkzeug | Flask 生态自带，安全可靠 |
| **容器化** | Docker | 作业要求的分发方式 |
| **云部署** | Render.com | 免费，无需信用卡，一键部署 |
| **CI** | GitHub Actions | 自动运行测试 + 构建镜像 |

**选型核心思路：** 以你现有的 Python 基础出发，选最简单、最不折腾的技术栈，把精力放在功能逻辑上。

## 10. 风险与未决问题

### 风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 项目功能较多，15天可能紧张 | 核心功能做不完 | 优先级管理：先保证核心任务 + 商城 + 家长端，宠物和勋章优先度递减 |
