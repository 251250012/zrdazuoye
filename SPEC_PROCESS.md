# SPEC_PROCESS.md · 规格说明产生过程

## 第一步：头脑风暴与需求收集

### 初始想法
用户希望构建一个亲子互动 Web 程序，帮助家长和孩子共同获得乐趣并解决生活问题。经过讨论，确定方向为"亲子任务打卡 & 奖励系统"。

### 关键问题与问答

1. **家长端是否需要独立？** → 需要，家长和孩子使用独立账号
2. **积分机制** → 采用盲盒机制，孩子只看到最终分数，不知道基准分
3. **宠物系统细节** → 5 种动物类型（小猫、小狗、小兔子、小恐龙、小企鹅），各有 3 种互动反应
4. **数学题分级** → 4 个年级（幼儿园到三年级），每个年级不同难度
5. **双货币系统** → 最初设计为积分（兑换活动）+ 金币（宠物系统），后简化为单金币

### 被否决的功能
- 社交功能：考虑到隐私和复杂度，暂不加入
- 实时成长时间：决定用操作次数代替真实时间

## 第二步：架构设计

### 确定方案
- 技术栈：Python Flask + SQLite + Jinja2
- 架构模式：服务端渲染，无前后端分离
- 数据模型：7 张表覆盖所有功能模块
- 用户系统：家长账号 + 孩子账号，独立登录

### 设计评审
用户逐项审批了以下内容：
1. 整体架构图 → ✅
2. 数据模型设计 → ✅
3. 页面结构 → ✅
4. 设置功能 → ✅

## 第三步：规格文档编写

根据审批通过的设计，编写 SPEC.md：
- 功能规格（5 大功能模块）
- 盲盒机制详细说明
- 宠物成长公式
- 数据模型定义（7 张表）
- 安全与部署要求

## 第四步：实现计划编写

根据 SPEC.md，拆分为 14 个实现任务：
1. 项目脚手架
2. 数据库 Schema 与模型
3. 认证系统
4. 家长任务管理
5. 孩子打卡（盲盒）
6. 活动管理 + 美好时刻
7. 商城与兑换
8. 宠物系统
9. 数学游戏
10. 勋章系统
11. 周报
12. 心愿单
13. 家长设置
14. Docker + CI + README

## 第五步：迭代变更

开发过程中根据用户反馈进行了多次重大调整：

1. **去掉心愿单** → 功能冗余，用户不需要
2. **去掉活动/商城系统** → 积分制简化，去掉兑换，仅保留金币
3. **宠物支持多只同时养** → 模板 bug 修复，最多 5 只
4. **宠物蛋收费** → 200 金币购买，无免费领养
5. **数学题独立为答题系统** → 每天 10 轮，每轮最多 10 题
6. **宠物图片** → 用户提供真实图片，每只宠物固定选图
7. **PWA 支持** → 可添加到手机桌面
8. **用户名规则** → 家长可重名，密码全系统唯一
9. **任务简化** → 波动自动 ±20%，家长只需设基准分

## 实现方式
采用 Subagent-Driven Development 流程：
- 每个任务由独立子代理实现
- 每个任务完成后进行代码审查
- 所有任务完成后进行整体审查
